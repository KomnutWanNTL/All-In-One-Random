# All-In-One Random

เว็บแอปแบบ static สำหรับสุ่มข้อมูลตัวอย่างภาษาไทย/สากล เช่น เลขบัตรประชาชน, บัตรเครดิต (mock), เบอร์โทร, ที่อยู่ไทย, อีเมล, บริษัท, และ UUID

## จุดประสงค์
- ใช้สร้าง mock data สำหรับเดโม, ทดสอบฟอร์ม, และทดสอบ UI
- ทำงานฝั่ง client ทั้งหมดผ่านไฟล์ JSON ในโปรเจกต์

## ฟีเจอร์ปัจจุบัน
- สุ่มเลขบัตรประชาชนไทยพร้อม checksum
- สุ่มบัตรเครดิต (ข้อมูลตัวอย่าง)
- สุ่มเบอร์โทรศัพท์มือถือไทย
- สุ่มที่อยู่ไทยจากชุดข้อมูลจังหวัด/อำเภอ/ตำบล
- สุ่มอีเมล
- สุ่มชื่อบริษัทและเว็บไซต์
- สุ่ม UUID v4-like

## โครงสร้างโปรเจกต์
- `index.html`: หน้าเว็บหลักและปุ่มสุ่ม
- `css/style.css`: custom style (ปัจจุบันว่าง)
- `js/main.js`: bootstrap app และ event เริ่มต้น
- `js/api.js`: โหลดข้อมูล JSON และเก็บ state รวมใน `appData`
- `js/generators.js`: ฟังก์ชันสุ่มข้อมูลแต่ละประเภท
- `js/ui.js`: ประกอบข้อมูลและ render การ์ด
- `src/*.json`: dataset สำหรับสุ่มข้อมูล

## Data Flow
1. หน้าเว็บโหลด `js/main.js`
2. `DOMContentLoaded` เรียก `loadAllData()`
3. `loadAllData()` fetch ไฟล์ใน `src/` แบบขนาน
4. เมื่อโหลดสำเร็จ ตั้งค่า `appData.isLoaded = true`
5. เรียก `displayRandomData()` เพื่อสุ่มและแสดงผล
6. ผู้ใช้กดปุ่ม "สุ่มข้อมูลใหม่" เพื่อ render รอบใหม่

## วิธีรันบนเครื่อง (Local)
1. เปิด terminal ที่โฟลเดอร์โปรเจกต์
2. รันคำสั่ง

```bash
python3 -m http.server 8000
```

3. เปิดเบราว์เซอร์ที่ `http://localhost:8000`

หมายเหตุ: ควรรันผ่าน static server เท่านั้น ไม่ควรเปิดแบบ `file://` เพราะมีการ `fetch()` ไฟล์ JSON

## Implementation Best Practices

### 1. แยกความรับผิดชอบของโมดูลให้ชัด
- ให้แต่ละไฟล์ทำหน้าที่เดียวตามโครงสร้างที่มีอยู่
- หลีกเลี่ยงการผสม logic โหลดข้อมูล + render UI ในไฟล์เดียว

### 2. จัดการข้อมูลด้วย state กลาง
- ใช้ `appData` เป็นแหล่งข้อมูลกลาง
- ตรวจ `isLoaded` ก่อนเรียก generator ที่ต้องใช้ dataset

### 3. ทำ Error Handling แบบระบุสาเหตุ
- เพิ่มการตรวจ `response.ok` ใน `loadAllData()`
- แสดงว่าไฟล์ใดโหลดไม่สำเร็จ เพื่อ debug ได้เร็วขึ้น

### 4. เพิ่มประสิทธิภาพ address generator
- สร้าง index map ล่วงหน้า:
  - `districtsByProvinceId`
  - `subDistrictsByDistrictId`
- ลดการ `filter()` ซ้ำบน array ขนาดใหญ่ทุกครั้งที่สุ่ม

### 5. หลีกเลี่ยง unbounded recursion
- ในการสุ่มที่อยู่ ควรใช้ loop พร้อมจำนวน retry สูงสุด (max attempts)
- ถ้าสุ่มไม่สำเร็จให้คืน fallback/null ที่ควบคุมได้

### 6. ปรับปรุงความถูกต้องของข้อมูลสุ่ม
- บัตรเครดิตควรสร้างหมายเลขที่ผ่าน Luhn algorithm
- ระบุใน UI/README ว่าเป็นข้อมูลจำลอง ไม่ใช่ข้อมูลธุรกรรมจริง

### 7. ปลอดภัยและ maintainable ในงาน UI
- สำหรับฟีเจอร์ใหม่ ใช้ `addEventListener` แทน inline `onclick`
- ลดการใช้ `innerHTML +=` ในลูป; พิจารณา `DocumentFragment` หรือประกอบ string แล้ว assign ครั้งเดียว

### 8. แนวทางทดสอบแบบง่ายหลังแก้โค้ด
- เปิดหน้าเว็บแล้วกดสุ่มหลายครั้ง
- ตรวจว่า card แสดงครบทุกประเภท
- ตรวจว่าไม่มี error ใน console
- ตรวจความสัมพันธ์ที่อยู่: จังหวัด -> อำเภอ -> ตำบล -> รหัสไปรษณีย์

## ข้อจำกัดที่ควรรู้
- เป็น client-side random generator เท่านั้น
- ความสุ่มของ `Math.random()` ไม่เหมาะกับ use case ความปลอดภัยสูง
- UUID ปัจจุบันเป็น v4-like สำหรับงาน mock

## Roadmap ที่แนะนำ
- เพิ่มปุ่มคัดลอกข้อมูลในแต่ละ card
- เพิ่มตัวเลือกประเภทข้อมูลที่จะสุ่ม
- เพิ่มการสุ่มใหม่เฉพาะ card
- เพิ่ม dark/light mode
- เพิ่ม dropdown เลือกจังหวัดก่อนสุ่มที่อยู่
